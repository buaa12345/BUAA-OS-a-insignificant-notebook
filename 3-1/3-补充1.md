
**例子**

    int g_init = 42;
    int g_uninit;
    const char msg[] = "Hi\n";

    int add(int a, int b) {
        return a + b + g_init;
    }

    int main() {
        g_uninit = add(1, 2);
        return msg[0];
    }

这几个对象分别很典型：

1. g_init 已初始化的全局变量，通常进 .data
2. g_uninit 未初始化的全局变量，通常进 .bss
3. msg 是只读常量，通常进 .rodata
4. add 和 main 是函数，通常进 .text

**第一步：先看 section 视角**

如果这是编译后的目标文件或可执行文件，从“链接”角度看，大致会有这些 section：

1. .text
   放 add 和 main 的机器指令

2. .rodata
   放字符串常量或只读常量
   这里的 "Hi\n" 通常在这里

3. .data
   放已初始化全局变量
   这里的 g_init = 42 在这里

4. .bss
   放未初始化全局变量
   这里的 g_uninit 在这里

可以把它想成：

    section 级别的分类
    .text    -> 代码
    .rodata  -> 只读常量
    .data    -> 已初始化数据
    .bss     -> 未初始化数据

这一步是“按内容类型分类”。

**第二步：再看 segment 视角**

当链接器把最终可执行文件做出来后，操作系统加载它时，通常不是一个 section 一个 section 地装，而是按 segment 装。

一个非常典型、简化后的 ELF 可执行文件，会有两个主要的可装载 segment：

1. 代码段 segment，权限通常是 读 + 执行
2. 数据段 segment，权限通常是 读 + 写

它们和 section 的关系通常是这样：

1. 第一个 segment，R-X
   常常包含 .text 和 .rodata
   有时还会顺带包含 ELF header、program header table 这些靠前内容

2. 第二个 segment，RW-
   常常包含 .data 和 .bss

也就是：

    segment 级别的归并
    R-X segment  -> .text + .rodata
    RW- segment  -> .data + .bss

这里你就能看出核心区别：

1. section 是细分后的“内容分类”
2. segment 是为了装入内存而进行的“打包”

**为什么 .text 和 .rodata 常放到同一个 segment**

因为它们有相近的运行时权限需求：

1. .text 要可执行，通常不可写
2. .rodata 只读，也不可写

所以链接器常把它们安排在同一个只读或读可执行的装载区域里。这样内存权限更好管理，页映射也更方便。

简化理解就是：

1. 不能写的东西放一堆
2. 需要写的东西放另一堆

所以：

1. .text、.rodata 常归到前面的只读/可执行 segment
2. .data、.bss 常归到后面的可写 segment

**为什么 .bss 很特别**

.bss 最能说明 section 和 segment 不完全是一个概念。

在文件里：

1. .bss 是一个 section
2. 但它通常几乎不占实际文件内容

在内存里：

1. 它必须真实存在
2. 而且初值要全为 0

所以对 RW- 这个数据 segment 来说，往往会出现：

$$
\text{内存大小} > \text{文件大小}
$$

因为：

1. .data 那部分要从文件里读出来
2. .bss 那部分不需要从文件里读，只要在内存里额外分配并清零即可

你可以把它想成：

    文件中的数据段内容
    [ .data ]

    加载到内存后的数据段内容
    [ .data ][ .bss(补零) ]

这就是为什么 program header 里会分别记录“文件中占多少”和“内存中占多少”。

**把这个程序画成一个简化布局**

先看文件内部，按 section 理解：

    ELF 文件
    +----------------------+
    | ELF header           |
    +----------------------+
    | Program header table |
    +----------------------+
    | .text                |  <- add, main 的机器指令
    +----------------------+
    | .rodata              |  <- "Hi\n"
    +----------------------+
    | .data                |  <- g_init = 42
    +----------------------+
    | .bss                 |  <- 逻辑上存在，文件里通常不真正占内容
    +----------------------+
    | Section header table |
    +----------------------+

再看加载到进程内存后，按 segment 理解：

    进程虚拟地址空间
    +-----------------------------------+
    | R-X LOAD segment                  |
    | ELF header / PHDR / .text /.rodata|
    +-----------------------------------+
    | RW- LOAD segment                  |
    | .data + .bss                      |
    +-----------------------------------+

注意，这里说的是“常见布局”，不是所有 ELF 都绝对一样，但操作系统课里这样理解是对的，而且最实用。

**把这个例子逐项对应起来**

对刚才那段代码：

1. add
   属于函数
   最后在 .text
   一般被放入 R-X segment

2. main
   也是函数
   最后在 .text
   一般被放入 R-X segment

3. msg
   是只读常量
   最后在 .rodata
   一般也被放入 R-X 或只读 segment

4. g_init
   已初始化全局变量
   最后在 .data
   一般被放入 RW- segment

5. g_uninit
   未初始化全局变量
   最后在 .bss
   一般也被放入 RW- segment

所以最终结果就是：

1. 代码和只读常量去前面的代码类 segment
2. 可写全局数据去后面的数据类 segment

**为什么不直接按 section 装载，而要有 segment**

因为 section 太细，太偏“文件组织”；操作系统加载程序时关心的是：

1. 哪一大片内容该映射到哪个虚拟地址
2. 那一大片内容该给什么权限
3. 哪部分需要从文件读
4. 哪部分只需要补零

这些都更适合用 segment 表达，而不是用 section。

换句话说：

1. section 更适合编译器、汇编器、链接器
2. segment 更适合加载器、操作系统、MMU 权限管理

**真实 ELF 里为什么有时会看到三个甚至更多 segment**

如果你看真正 Linux 下的 ELF，可执行文件往往不止两个 segment，还可能看到：

1. PT_INTERP
   指向动态加载器路径

2. PT_DYNAMIC
   动态链接相关信息

3. PT_NOTE
   一些附加说明

4. GNU_RELRO、GNU_STACK
   安全相关属性

但在操作系统课程入门阶段，你先抓住最重要的两个 LOAD segment 就够了：

1. 一个负责代码和只读内容
2. 一个负责可写数据和 bss

**一句话总结这个例子**

对于这段小程序：

1. .text、.rodata 是 section 层面的“内容分类”
2. 它们通常一起进入前面的只读/可执行 segment
3. .data、.bss 是 section 层面的“数据分类”
4. 它们通常一起进入后面的可写 segment
5. .bss 在文件里几乎不占内容，但在内存里必须分配并清零
